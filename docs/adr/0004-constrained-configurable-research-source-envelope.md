# Constrained Envelope For User-Configured Research Source Definitions

The Configurable Research Source Envelope intentionally limits what shape of remote API the app user can register. The supported surface is JSON over HTTP using GET or POST, a single API token carried in either a request header or a query parameter, response items reachable through a fixed dot-and-bracket-index path with `field_map.content` resolved as an array of paths joined by newline, and at most one of page-number, offset, or cursor pagination capped at five pages per source per research job. The envelope explicitly excludes OAuth flows, request signing such as HMAC, non-JSON response bodies, multipart or streaming requests, user-supplied pre-request or post-response scripts, and per-request rate-limit declarations beyond a static `max_parallel` integer. The decision is made now because the envelope shape is the union of the storage format, the Research Source Panel form, the request execution layer, and the security model; broadening it after launch would touch every layer.

**Consequences**

The Research Source Panel form has a known finite shape: one URL field, method toggle, header and query and body template fields with `{token}` / `{query}` / `{page}` / `{page_size}` / `{cursor}` placeholders, `field_map` editor for `url` / `title` / `content` paths with `content` as an array, pagination block with one of three modes, and one credential field. No script editor, no auth-flow wizard, no custom header marshalling beyond static templates.

Anything an upstream API requires that is outside this envelope is treated as out of scope for User-Configured Research Source. A user whose target API needs OAuth or HMAC cannot register it through this path; they must wait for a future ADR or, for high-value cases, propose the API as a candidate Built-in Research Source. The envelope is deliberately small, not aspirationally complete.

Pagination is bounded by two stacked caps: each Research Source declares its own `max_pages` (≤ 5), and a global `MAX_PAGES_PER_SOURCE = 5` constant in the Researcher Tool Backend rejects any definition whose declared max is higher at save time. This double cap protects against both user misconfiguration and future changes that might silently relax the per-source declaration.

Credential storage holds at most one secret per Research Source, masked as front-zero-back-four (e.g., `***f456`) when returned to the Anna App Shell. The credential never travels through job records, log lines, frontend bundle assets, or fixture files. Built-in Research Source entries also use the unified credential storage so the masking rule applies identically to Tavily.

The HTTP execution layer issues one retry with 1 s backoff for GET requests against 429 or 5xx status, and no retry for POST or other 4xx classes. Failures are classified into six fixed error codes — `auth_failed | rate_limited | upstream_5xx | timeout | bad_definition | empty_result` — where `empty_result` is not treated as an error and flows through the normal path. The Research Step Decision prompt lists the semantic of each code and the recommended LLM response, so the Iterative Research Loop can route the next iteration deterministically.

Tests must lock the envelope boundary: definitions outside the envelope (e.g., declaring `max_pages: 10`, missing token field, multipart content type, OAuth-shaped auth block) must be rejected at save time; valid definitions covering all three pagination modes against fake HTTP fixtures must execute end-to-end through the Lexical Context Selector.
